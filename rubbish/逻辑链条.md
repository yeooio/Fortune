# 逻辑链条

## 1. 边界突破：SSRF 盲读与探测

- 技术实现：利用 /api/avatar/download 接口未对协议头和目标地址进行过滤的漏洞。
- 严谨动作：攻击者首先使用 file:// 协议读取 /etc/redis/redis.conf 配置文件。
- 目的：获取 Redis 的认证密码（requirepass）、绑定地址，以及数据存储目录（dir）。

## 2. 配置解析：攻击参数初始化

- 技术实现：脚本中的 parse_redis_config 函数对读取到的配置文本进行解析。
- 严谨动作：提取出后续 CRLF 注入所需的凭证和路径信息。
- 重要性：没有这一步，后续的 Redis 命令注入将因缺乏认证或路径错误而失败。

## 3. 协议走私：CRLF 注入控制数据库

- 技术实现：利用 urllib 等库对换行符（%0d%0a）解析不严的漏洞，在 SSRF 的 URL 中嵌入 Redis 指令。
- 严谨动作：发送 AUTH 进行认证，随后发送 SAVE 命令。
- 目的：强迫 Redis 将包含敏感密钥的内存数据持久化到磁盘上的 dump.rdb 文件中。

## 4. 凭证窃取：Secret Key 提取

- 技术实现：通过 SSRF 再次使用 file:// 协议读取刚刚生成的 dump.rdb 二进制文件。
- 严谨动作：脚本通过正则表达式从二进制流中匹配出 64 位十六进制字符串（Flask 的 secret_key）。
- 实质：拿到了 Flask 应用签发身份证明的最高权限公章。

## 5. 权限提升：Session 伪造与验证

- 技术实现：在本地使用 itsdangerous 库配合窃取的 secret_key 对自定义 Session 数据进行签名。
- 严谨动作：伪造一个 is_admin: True 的 Cookie，并通过访问 /admin/users 接口验证权限是否生效。
- 结果：攻击者从普通用户成功转变为系统管理员。

## 6. 漏洞预埋：Pickle 负载注入

- 技术实现：再次利用 CRLF 注入漏洞，向 Redis 写入一个经过 Base64 编码的恶意 Pickle 序列化对象。
- 严谨动作：将该对象存储在服务器后台会读取的特定键值（如 online_user:attacker）中。
- 实质：在服务器端预埋可触发的恶意负载。

## 7. 夺取大脑：反序列化 RCE 触发

- 技术实现：管理员访问特定后台页面（如 /admin/online_users），触发服务器对 Redis 数据的读取。
- 严谨动作：服务器执行 pickle.loads() 还原对象时，自动运行 Payload 中 __reduce__ 方法定义的系统命令。
- 目的：获得远程代码执行（RCE）权限，通常表现为反弹 Shell 或写入执行脚本。

## 8. 横向移动：内网 MCP 服务攻陷

- 技术实现：利用 RCE 权限在服务器上运行 Python 攻击脚本，探测内网 3000/8080 等端口。
- 严谨动作：直接向内部 MCP 服务发送 execute_command 指令。
- 终局：彻底控制服务器核心管理组件，实现对整个系统的最高权限接管。
